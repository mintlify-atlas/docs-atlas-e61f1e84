The React Native Directory separates hand-edited data from generated data. You only ever edit `react-native-libraries.json`. Everything else is produced by scripts.

## The data pipeline

```
react-native-libraries.json
        │
        ▼
 bun data:update
        │
        ▼
  assets/data.json
  assets/check-data.json
        │
        ▼
   Next.js build
        │
        ▼
 reactnative.directory
```

1. **`react-native-libraries.json`** — the single source of truth for library entries. This is the only file contributors edit.
2. **`bun data:update`** — fetches live data and writes the generated asset files.
3. **`assets/data.json`** — the enriched dataset consumed by the site and the public API.

## What `bun data:update` does

Running `bun data:update` performs the following steps for every library entry:

<Steps>
  <Step title="Fetch GitHub data">
    For each entry, the script reads the repository metadata from the GitHub API: name, description, topics, star count, fork count, open issue count, last push date, license, and whether the repository is archived.

    For monorepo entries (where `githubUrl` points to a subdirectory), the name, description, homepage, and topics are extracted from the `package.json` in that subdirectory. GitHub stats (stars, forks, issues) are always based on the root repository.
  </Step>
  <Step title="Fetch npm data">
    The script fetches download counts and package metadata from the npm registry and npm-stat API.
  </Step>
  <Step title="Auto-detect fields">
    Several fields are detected automatically from the package contents:

    - **`newArchitecture`** — inferred from the module type (Expo module, Nitro module, or Turbo module) found in the package source
    - **`configPlugin`** — detected if the package contains an Expo config plugin
    - **`unmaintained`** — set automatically if the GitHub repository is archived or has not received a push in more than two years

    If auto-detection produces a result, you do not need to set these fields manually in `react-native-libraries.json`.
  </Step>
  <Step title="Calculate scores">
    The script runs `calculate-score.ts` to compute the directory score for each library, and a separate popularity score based on download counts and star counts.
  </Step>
  <Step title="Write output files">
    The enriched data is written to `assets/data.json` and `assets/check-data.json`.
  </Step>
</Steps>

## Running `bun data:update` locally

The script requires a GitHub personal access token to avoid rate limiting:

```sh
GITHUB_TOKEN=<your-token> bun data:update
```

Get a token at [github.com/settings/developers](https://github.com/settings/developers). The callback URL does not matter for this use case.

<Note>
  By default, `bun data:update` uploads the result to a Vercel Blob store. For local debugging, set `ONLY_WRITE_LOCAL_DATA_FILE` to `true` inside `scripts/build-and-score-data.ts` to skip the upload and write only to the local `assets/data.json` file instead.
</Note>

## Files you must not edit manually

The following files are generated outputs. Do not edit them by hand — your changes will be overwritten the next time the data pipeline runs.

| File | Generated by |
|---|---|
| `assets/data.json` | `bun data:update` |
| `assets/check-data.json` | `bun data:update` |
| `public/llms.txt` | `bun llms:generate` |
| `public/llms-full.txt` | `bun llms:generate` |

## How changes reach production

Commits to the `main` branch are deployed automatically via Vercel. You do not need to run `bun data:update` yourself when submitting a library entry — the production build runs the full data pipeline with its own `GITHUB_TOKEN`.

```sh
# Only needed if you are deploying your own fork
bunx vercel
```
